# Restore accidentally deleted worktree folder

## Problem

When a user manually deletes a worktree folder (e.g. via `rm -rf`), the git worktree reference still exists but the working directory is gone. The runner currently has no way to recover from this state.

## Proposed Solution

Add a restore/repair command (or integrate into an existing command) that detects orphaned worktree references — where the git worktree entry exists but the path on disk is missing — and recreates the working directory by checking out the branch again.

## Expected Behavior

- Runner detects that the worktree path no longer exists on disk
- Runner offers to restore it (re-checkout the branch into the original path)
- Restored worktree is usable without any manual git commands

## Current Behavior

The runner has no mechanism to detect or recover from a manually deleted worktree folder. The registry entry and git worktree ref remain, but the directory is gone.

## Additional Context

The `prune` command already skips entries whose `path` no longer exists (`r.path.exists()`), so the registry is self-healing in one direction. Restoration would be the reverse: instead of pruning the record, recreate the missing directory.
