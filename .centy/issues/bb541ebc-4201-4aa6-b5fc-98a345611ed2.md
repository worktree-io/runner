# Add config to control editor foreground/background in terminal sessions

## Summary

Add a config option (e.g. `editor.background`) that lets the user choose whether the editor opens in the foreground (blocking the terminal until closed) or only in the background (fire-and-forget). This setting is only meaningful in terminal sessions — IDE integrations manage their own window lifecycle and are unaffected.

## Motivation

- When running `runner open` from a terminal, some users want the terminal to wait until the editor is closed (foreground, useful for shell workflows and hooks that depend on the editor finishing)
- Other users prefer the editor to open in the background so they can continue using the terminal immediately
- Currently there is no way to control this; the behavior is implicit and depends on how the editor is launched

## Proposed Change

```toml
[editor]
background = true  # default: false
```

When `background = true`, the editor process is spawned without waiting (e.g. `Command::new(...).spawn()` instead of `...status()`).  
When `false` or unset, the terminal blocks until the editor exits (current behavior for terminal sessions).

This flag is ignored when the runner detects an IDE environment (e.g. VS Code, JetBrains), since those editors always return immediately.

## Affected Code

- `src/config/mod.rs` — add `background: bool` to `EditorConfig` (with `skip_serializing_if` so it is omitted when false)
- `src/opener/mod.rs` (or equivalent) — branch on `editor.background` to either `spawn()` or `status()` when launching the editor in terminal mode
