# Allow opening a worktree from inside an existing repo

Currently `worktree open` requires an explicit issue reference (`<REF>`) — a GitHub URL, `worktree://` deep link, or `owner/repo#N`.

## Proposal

Allow invoking `worktree open` **from inside a git repository** with no arguments. When no `<REF>` is provided and the current directory is inside a git repo, the CLI should:

1. Detect the repository root and remote origin
1. Derive the owner/repo from the remote URL
1. Generate a random adhoc branch name and create a new worktree

This enables a workflow where a user `cd`s into a cloned repo and runs `worktree open` to spin up a fresh worktree workspace without needing to look up the issue URL.

## Acceptance criteria

* [x] `worktree open` with no `<REF>` inside a git repo detects the repo and opens the worktree
* [x] Works with both HTTPS and SSH remote URLs
* [x] Shows a clear error when run outside a git repo without a `<REF>`
* [x] Existing `<REF>`-based behavior is unchanged
* [x] Generates a random adhoc name (not the current branch name)
