# fix: terminal loses worktree cwd when hook is interrupted with ^C

## Summary

When a hook script is opened in a terminal emulator, the bootstrap script uses:

```sh
#!/bin/sh
cd '/path/to/worktree'
<init_script>
exec "${SHELL:-sh}"
```

If the user presses `^C` during the hook (e.g. during a long `cargo build`), SIGINT kills the script before `exec` can run. Control returns to the parent login shell which is still at `~`, leaving the user in the wrong directory.

In the success case the `exec` does run and the user lands at the worktree path correctly.

## Root Cause

The `cd` is scoped to the bootstrap script's process. The `exec "${SHELL:-sh}"` at the end replaces that process with an interactive shell that inherits the correct cwd. But SIGINT interrupts the script before `exec` is reached, so the parent shell (at `~`) resumes instead.

## Proposed Fix

Add a SIGINT trap in the bootstrap script so that even when the hook is interrupted, `exec` still runs and the user lands at the worktree path:

```sh
#!/bin/sh
cd '/path/to/worktree'
trap 'exec "${SHELL:-sh}"' INT
<init_script>
exec "${SHELL:-sh}"
```

This is OS-agnostic — the trap runs in `/bin/sh` which is available on all Unix/macOS/WSL systems. On Windows the `wt` path already uses a separate mechanism and is unaffected.

## Affected File

`src/opener/terminal.rs` — the `bootstrap` format string (line 25).
