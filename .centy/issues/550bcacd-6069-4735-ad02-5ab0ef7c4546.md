# Per-repo hooks: support hook configuration scoped to individual repositories

## Overview

Currently hooks (`pre:open`, `post:open`) are configured globally in the runner config. Add support for per-repo hook configuration so that hooks can be defined and run in the context of a specific repository rather than globally.

## Motivation

Different repositories have different setup needs (e.g., one may need `npm install`, another `cargo build`). Per-repo hooks allow each repo to declare its own lifecycle scripts independently of the global config.

This is also a prerequisite for the multi-repo grouped workspace feature (#31), where per-repo hooks run for each individual worktree within the group.

## Proposed Behavior

- Per-repo hook config lives alongside the repo (e.g., in a `.worktree.toml` or equivalent file committed to the repo, or in the global config keyed by repo identifier)
- Supported hook points mirror the global ones: `pre:open`, `post:open`
- Per-repo hooks run in the context of that repo's worktree directory
- When a repo defines a hook, it also declares how it relates to the global hook via an enum:
  - `before` — run the per-repo hook first, then the global hook
  - `after` — run the global hook first, then the per-repo hook
  - `replace` — disable the global hook entirely, run only the per-repo hook

## Open Questions

- Where does per-repo config live? Options:
  - A dotfile in the repo (e.g., `.worktree.toml`) — portable, version-controlled
  - A section in the global `~/.config/runner/config.toml` keyed by repo remote URL or name
  - Both, with a defined precedence
- How does this interact with the Mustache templating already supported for hook scripts?
