# Init worktree.toml: scaffold commented template when file is missing

## Overview

When the runner operates on a repo (e.g., during `open`) and no `.worktree.toml` file exists at the repo root, automatically create an empty scaffold with inline comments that document all supported fields. This acts as a self-documenting starting point so contributors can enable per-repo hooks without consulting external docs.

## Motivation

`.worktree.toml` is an opt-in config file that most repos will not have initially. Without tooling support, users must know to create it by hand and remember the exact key names and allowed values. A generated scaffold lowers the barrier to adoption and serves as living documentation embedded in the repo.

## Proposed Behavior

- On first `open` for a repo (or via a dedicated `init` subcommand), if `.worktree.toml` is absent, write it to the repo root
- The generated file is purely comments — no active configuration — so it has no effect until the user edits it
- Comments document every supported field with its type, allowed values, and a short description:

```toml
# .worktree.toml — per-repo worktree configuration
# Commit this file to version-control to share settings with your team.

# [hooks]
# Configure lifecycle hooks that run when a worktree for this repo is opened.
# Each hook is a shell command (string) executed in the worktree directory.
# Mustache templating is supported (same variables as global hooks).
#
# Hook ordering controls how a per-repo hook interacts with the global hook
# configured in the runner's main config. Allowed values:
#   "before"  — run the per-repo hook first, then the global hook
#   "after"   — run the global hook first, then the per-repo hook
#   "replace" — skip the global hook entirely, run only the per-repo hook
#
# "pre:open" = "cargo build"
# "pre:open:order" = "after"   # default: "after"
#
# "post:open" = "npm install"
# "post:open:order" = "after"  # default: "after"
```

- The file is written only if it does not already exist (never overwrites)
- A short log line is emitted: `created .worktree.toml (no active config — edit to enable hooks)`

## Open Questions

- Should init happen automatically on `open`, or only via an explicit `runner init` / `runner config init` subcommand?
- Should the generated file include a `[hooks]` header (uncommented) with all keys commented out, or be entirely freeform comments?
