# Windows support gaps

## Current State

Windows binaries are built and released (MSVC x86_64), URL scheme registration works via the registry, and editor detection works via PATH lookup. However, several core features are broken or missing on Windows.

## Broken

### Hook execution (`src/hooks.rs`)
- Script permissions set via `#[cfg(unix)]` `chmod 0o755` — no Windows equivalent
- Hooks spawned via `Command::new("sh")` with `/bin/sh` hardcoded — does not exist on Windows
- Users with pre/post hooks configured will get silent failures

### Terminal integration (`src/opener/terminal.rs`)
- Bootstrap script uses Unix shell syntax (`exec "${SHELL:-sh}"`)
- File permissions set via `#[cfg(unix)]` only
- Windows Terminal (`wt`) is recognized in editor fallback and setup detection but never integrated into `try_terminal_with_init`

### PATH augmentation (`src/opener/shell.rs`)
- Hardcoded Unix paths (`/usr/local/bin`, `/opt/homebrew/bin`, `/opt/homebrew/sbin`)
- Uses `:` as PATH separator instead of `;`

### Auto-terminal detection (`src/opener/mod.rs`)
- `app_exists()` checks `/Applications/` (macOS-only)
- `open_hook_in_auto_terminal()` only tries macOS apps, no Windows Terminal fallback

## What Would Be Needed

- Hook execution: write `.bat` or `.ps1` scripts on Windows, spawn via `cmd /C` or `powershell -File`
- Hook permissions: skip `chmod` on Windows (scripts are executable by default)
- Terminal spawning: add Windows Terminal (`wt -d .`) to `try_terminal_with_init`
- PATH augmentation: use `;` separator and skip homebrew paths on Windows
- Document current limitations in README until the above is implemented
