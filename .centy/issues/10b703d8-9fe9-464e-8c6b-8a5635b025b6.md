# open-multi command: open one issue across multiple repos in a unified workspace

## Overview

Add a new `worktree open-multi` command that takes repositories and issue references, then opens all of them as worktrees under a single unified workspace folder.

## Motivation

Users working on an issue that spans multiple related repos (e.g., backend + frontend, or microservices) currently have to run `worktree open` separately per repo and open each folder individually. `open-multi` solves this by creating one shared workspace with all repos checked out together.

## Command Signature

Two modes are supported:

### 1. One issue for all repos

```
worktree open-multi <issue> <owner/repo-a> <owner/repo-b> [...]
```

The same issue/branch is used for every repo.

```
worktree open-multi worktree-io/runner#42 worktree-io/runner worktree-io/landing
worktree open-multi https://github.com/acme/backend/issues/7 acme/backend acme/frontend
```

### 2. One issue per repo

```
worktree open-multi <owner/repo-a>#<issue-a> <owner/repo-b>#<issue-b> [...]
```

Each repo gets its own issue reference, which may differ between repos.

```
worktree open-multi acme/backend#7 acme/frontend#12 acme/infra#3
```

## Behavior

- Creates a unified workspace folder with a random human-friendly name (e.g., `~/workspaces/bold_turing/`)
- For each repo, clones (bare) if not already cached, then creates a worktree at `<workspace>/<repo-name>-<issue-number>/` (e.g., `runner-42/`); if no issue is associated with that repo, the folder is just `<repo-name>/`
- Branch name is derived from the issue for each repo (e.g., `issue-42`) and **always created in every repo** regardless of whether the issue exists there
- Editor is launched at the unified workspace root (not per-repo)
- TTL applies to the entire unified folder
- Per-repo hooks (`.worktree.toml`) run for each repo; group-level hooks (global config) run once for the workspace
- Hook ordering in per-repo config is declared via an enum: `before` (per-repo first, then global), `after` (global first, then per-repo), or `replace` (per-repo only)

## Workspace Layout

```
~/workspaces/bold_turing/
  runner-42/    ← worktree for worktree-io/runner at issue-42
  landing-12/   ← worktree for worktree-io/landing at issue-12
```

In mode 1 (one issue for all repos):

```
~/workspaces/bold_turing/
  runner-42/    ← worktree for worktree-io/runner at issue-42
  landing-42/   ← worktree for worktree-io/landing at issue-42
```

## Open Design Questions

### 1. Workspace naming
The spec calls for a "random human-friendly name" (`bold_turing`). What is the source corpus?
Options: adjective+scientist surname from a bundled list, OS word list, external crate, or something else.

### 2. Workspace path type
`Workspace::open_or_create` encodes the issue identity in the path via `issue.temp_path()`.
For open-multi the root is independent of any single issue. Options:
- Create a new `MultiWorkspace` type with its own path/naming logic
- Add a `Workspace::within_root(dir)` constructor
- Reuse `Workspace` and discard its computed path (fragile)

### 3. TTL registration granularity
Today each worktree is registered individually. For a multi-workspace, should we register:
- The **workspace root** as one entry (prune all subdirectories together)?
- Each subdirectory separately (prune independently)?
- A new registry entry type that tracks a root + its children?

### 4. Parsing ambiguity in mode 1
`worktree open-multi acme/api#42 acme/web` — the parser must distinguish "same issue #42 applied to `acme/web`" from "context-only checkout of `acme/web` on its default branch."
Proposed rule: if the first argument carries a `#`-issue and subsequent arguments do not, treat the issue from the first arg as the shared issue; bare repo slugs are context-only.
Is mixed mode (`acme/api#42 acme/web acme/backend#7`) allowed or a parse error?

### 5. Hook execution order
"Per-repo hooks run for each repo; group-level hooks run once for the workspace." Needs a decision on sequencing:
- Global `pre:open` runs once **before** all repos, then per-repo `pre:open` for each?
- Or interleaved (per-repo pre → open → per-repo post, then global post once)?

### 6. `--print-path` semantics
For `open-multi --print-path`, should the printed path be the unified workspace root or the list of per-repo subdirectory paths?

### 7. Idempotency / re-opening
Workspace names are random, so a second invocation of the same `open-multi` args would create a new root. Options:
- Derive the workspace name deterministically from the sorted set of `repo#issue` pairs (e.g., hash → adjective+noun)
- Accept that re-running always creates a fresh workspace (document this)
- Scan the TTL registry for an existing entry that matches the same repo+issue set
