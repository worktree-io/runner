# feat: worktree issue create — create a new Centy issue and open its workspace

## Feature request

\`worktree open centy:<n>\` can open an existing Centy issue as a worktree, but there is no way to create a new issue from the CLI and immediately land in its workspace.

## Proposed command

\`worktree issue create [--title <title>] [--body <body>]\`

Workflow:

1. Shell out to the \`centy\` CLI to create a new issue and capture the assigned issue number.
2. Construct the shorthand \`centy:<n>\` and call the existing \`worktree open\` path — reusing \`parse_centy\`, workspace creation, hook execution, and editor launch as-is.

## Why this belongs in \`worktree\`

Users who work exclusively with local Centy projects today have to:
1. \`centy issue create …\`
2. Note the returned issue number
3. \`worktree open centy:<n>\`

A single \`worktree issue create\` call removes that context-switch and keeps the create→workspace transition atomic (pre/post hooks run automatically).

## Acceptance criteria

- \`worktree issue create\` shells out to the \`centy\` binary, errors clearly if it is not in PATH.
- On success, behaves identically to \`worktree open centy:<n>\` for the new issue.
- \`--no-hooks\` flag forwarded from \`worktree open\`.
- Works from any directory inside a Centy project (\`.centy/\` ancestor detection reused from \`find_centy_root\`).
