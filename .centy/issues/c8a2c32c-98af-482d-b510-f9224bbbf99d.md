# macOS URL scheme registration resets after a day

## Summary

After running `runner scheme install`, the `worktree://` URL scheme stops working after roughly a day. macOS periodically rebuilds its Launch Services database, wiping custom scheme registrations, and nothing re-registers during a live session.

## Root Cause

Two bugs in the install flow:

1. **`launchctl load` is never called** after writing the plist to `~/Library/LaunchAgents/` — the agent only activates at the next login, leaving the current session unprotected immediately after install.
2. **No `StartInterval`** in the launch agent plist — even once loaded, `lsregister` only runs at login, not periodically. When macOS rebuilds the LS database mid-session the scheme is lost until reboot.

## Fix

- Call `launchctl bootstrap gui/$(id -u) <plist>` immediately after writing the plist in `install_launch_agent`.
- Add a `StartInterval` key (e.g. 3600 seconds) to the plist so `lsregister -f` re-runs hourly and recovers from LS database resets without requiring a logout.

## Acceptance Criteria

- [ ] Launch agent is loaded immediately on `runner scheme install` without requiring a logout
- [ ] Launch agent plist includes `StartInterval` so registration is refreshed periodically
- [ ] `runner scheme status` reflects the registered state correctly after the fix
