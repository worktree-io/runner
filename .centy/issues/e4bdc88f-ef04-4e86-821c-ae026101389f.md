# Multi-repo workspace: open multiple repositories under a single unified folder

## Overview

Add an option to register multiple repositories together so that all of them point to worktrees under a single unified parent folder. When the user opens an issue, the editor is launched in that unified folder rather than in an individual repository root.

## Motivation

Users working across multiple related repositories (e.g., a monorepo split or microservices) want a single editor window that spans all of them, with worktrees from each repo checked out as subdirectories of one shared workspace folder.

## Proposed Behavior

- Allow configuring a named group of repositories
- A subcommand lists all configured repo groups so the user can pick which one to open
- The unified workspace folder is assigned a random human-friendly name (like Docker container names, e.g., `happy_torvalds`, `bold_turing`) instead of being derived from the issue identifier
- When opening, each repo creates a worktree under a shared parent directory (e.g., `~/workspaces/<random-name>/`)
- The editor is launched at the shared parent folder root
- The unified folder layout could be:
  ```
  ~/workspaces/bold_turing/
    repo-a/   ← worktree for repo-a
    repo-b/   ← worktree for repo-b
    repo-c/   ← worktree for repo-c
  ```
- The workspace TTL applies to the entire group: the unified folder (and all its repo worktrees) expires together based on a single TTL timer
- Both group-level and per-repo hooks are supported:
  - Group hooks run once for the unified folder (e.g., open the editor)
  - Per-repo hooks run for each individual repo worktree (e.g., install dependencies, run build); see #32 for details
    - Configured via a `.worktree.toml` file committed to the repo root — portable and version-controlled
    - Hook ordering is declared in the per-repo config via an enum: `before` (per-repo first, then global), `after` (global first, then per-repo), or `replace` (per-repo only)
    - Mustache templating applies to per-repo hook scripts the same way it does for global hooks

## Open Questions

- How should the group be configured? (config file, CLI flags, or a dedicated subcommand?)
- What happens if one repo does not have the branch/issue?
