# tmux as editor: session creation only works when a post:open hook is configured

## Bug

The tmux-specific session/window creation logic lives exclusively inside `try_terminal_with_init` (`src/opener/terminal.rs:78-91`), which is only reachable via `open_with_hook`. That function is only called when **both** an editor command **and** a post:open hook are configured (`commands/open/mod.rs:78-83`).

When tmux is configured as the editor but **no post:open hook exists**, the code falls into the `(Some(cmd), None)` branch and calls `open_in_editor` instead. That function treats tmux like any IDE and produces:

```
tmux /path/to/worktree
```

…which is invalid tmux syntax and fails silently.

### Bug 2 — `open_hook_in_auto_terminal` ignores tmux

`open_hook_in_auto_terminal` (`src/opener/mod.rs:26-50`) is the fallback that runs a post:open hook for IDE users. It only scans GUI terminals (iTerm, Warp, Ghostty, Terminal.app) — never tmux. So if you have an IDE configured and a post:open hook, the hook always runs inside Terminal.app even when tmux is available.

## Expected behavior

- Configuring `command = "tmux"` should always use the `new-session` / `new-window` logic, regardless of whether a post:open hook is present.
- `open_hook_in_auto_terminal` should check for a running tmux server (or tmux in PATH) and prefer spawning a new window there before falling back to GUI terminals.

## Steps to reproduce

1. `worktree setup` → choose tmux, no post:open hook.
2. `worktree open <issue>` → instead of a new tmux session/window, the command silently fails.
